# AGENTS.md

대한민국 국회의원 정보 플랫폼 개발을 위한 에이전트 운영 규칙.

이 문서는 **확장에 열린(open) 운영 헌장**이다.
특정 MVP의 산출물이나 범위를 못 박지 않는다. 대신 프로젝트가 어디까지 확장되더라도
변하지 않는 **원칙**과 **일하는 방식**을 정의한다. 기획 범위 밖의 새로운 아이디어·기능이
제안되더라도, 그것이 0장의 원칙을 지키는 한 이 문서의 운영 방식 안에서 다룬다.

제품 기획의 단일 진실 공급원은 `DOCUMENT.md`다. 단, `DOCUMENT.md`는 현재 시점의 범위를 담은
**스냅샷**이고, 이 문서는 시점과 무관하게 적용되는 **상위 규칙**이다.
새 아이디어가 `DOCUMENT.md`의 범위를 넘어서는 것은 막지 않는다. 막는 것은 0장 원칙 위반뿐이다.

---

## 0. 불변 원칙 (Invariants)

프로젝트가 어떻게 확장되든 아래는 변하지 않는다. 이것이 이 서비스의 정체성이다.
새 기능·아이디어는 이 원칙을 **만족하는 한** 자유롭게 추가될 수 있고,
**위반하면** 아무리 매력적이어도 채택하지 않는다.

1. **AI는 사실을 생성하지 않는다.** LLM은 다음 성격의 일만 한다: 개체 정합, 불일치 분류,
   출처 인용 응답(RAG). 요약·서술·평가·추론으로 새 사실을 만들어내지 않는다.
   — *새 AI 기능을 추가할 때의 판정 기준: "이 기능이 없던 사실을 만들어내는가?" 그렇다면 거부.*
2. **출처 없는 데이터는 노출하지 않는다.** 표시되는 모든 사실은 출처 메타데이터
   (`source_url`, `source_org`, `fetched_at` 등)를 동반한다.
3. **모르면 모른다고 한다.** 인용할 근거가 없으면 추측하지 않고 "관련 자료 없음"으로 응답한다.
4. **불일치는 병합하지 않고 드러낸다.** 출처 간 값이 다르면 고르거나 합치지 않고
   모든 출처를 함께 제시한다. 판단은 이용자 몫이다.
5. **성격이 다른 데이터는 섞지 않는다.** "검증이 필요한 사실"과 "참고용 외부 콘텐츠"를 분리한다.
   (현재: 교차검증 데이터 / 뉴스 피드. 향후 새 데이터 종류가 생겨도 이 분리 원칙을 적용한다.)
6. **타인의 콘텐츠를 재호스팅하지 않는다.** 외부 원문·이미지는 링크/임베드/참조만 한다.
7. **공개 대상은 공인의 공적 정보뿐이다.** 사적 정보는 다루지 않는다.
8. **객관성은 주장이 아니라 증명이다.** 코드·데이터·출처·검증 과정을 공개 가능한 상태로 유지한다.
   (오픈소스 전제: 코드 MIT/Apache 2.0, 데이터 CC BY 등 호환 라이선스.)

> 판단이 서지 않으면 진행하지 말고 결정 사항을 표면화하여 사람이 판단하게 한다.
> 원칙은 가정(assumption)으로 완화하지 않는다.

## 1. 확장 아이디어를 다루는 방식 (Open Intake)

이 프로젝트는 닫혀 있지 않다. 기획 범위 밖의 아이디어가 언제든 들어올 수 있고, 환영한다.
새 아이디어·기능 제안이 들어오면 다음 게이트를 거친다.

1. **원칙 적합성** — 0장 불변 원칙을 위반하는가? 위반하면 거부하고 이유를 남긴다.
   위반하지 않으면 통과(확장은 기본적으로 허용된다).
2. **분류** — 이 아이디어가 어디에 속하는지 정한다: 새 데이터 종류 / 새 화면 / 새 AI 활용 /
   새 데이터 소스 / 운영·배포 개선 등. 어디에도 안 맞으면 새 범주를 만든다.
3. **영향 범위** — 기존 데이터 모델·인터페이스에 미치는 영향을 적는다.
4. **편입** — 통과한 아이디어는 작업 루프(2장)에 태워 점진적으로 구현한다.
   필요하면 `DOCUMENT.md`에 새 범위로 반영을 제안한다(이 문서는 수정하지 않아도 된다).

> 원칙: **확장은 기본 허용, 원칙 위반만 차단.** 범위는 늘어날 수 있는 것으로 본다.

## 2. 에이전트 팀 (최대 3)

| 에이전트 | 역할 | 책임 |
| --- | --- | --- |
| **Architect** | 설계 | 구조·인터페이스 설계, 결정 근거(ADR) 기록, 작업 분해, 확장 아이디어의 영향 분석 |
| **Implementer** | 구현 | 설계·계획에 따른 작성, 검토 지적 반영 |
| **Reviewer** | 검토 | 0장 원칙 준수 검증, 정확성·엣지케이스 점검, 수정 지시 |

역할은 분리한다. 같은 에이전트가 자기 결과물을 스스로 통과시키지 않는다.
Reviewer는 어떤 작업이든 **0장 원칙 준수**를 최우선으로 본다.

---

## 3. 작업 루프

모든 작업은 다음을 반복한다. 큰 것을 한 번에 끝내려 하지 않고 점진적으로 키운다.

```
설계(Architect) → 계획(Architect→Implementer) → 구현(Implementer) → 검토(Reviewer) → 수정(Implementer)
```

- 중대한 원칙 위반이 발견되면 통과 금지. 수정 후 재검토한다.
- 각 반복 끝에 Reviewer가 **확정된 것 / 남은 것 / 다음 반복 권고**를 요약한다.
- 이 루프는 MVP든 그 이후의 확장이든 동일하게 적용된다.

---

## 4. 작동 방식 지침 (시점 무관)

아래는 산출물 목록이 아니라, **무엇을 만들든 적용되는 작업 습관**이다.

- **출처 결합 보관** — 사실 값은 항상 출처 메타데이터와 묶어 저장한다. 값만 단독 저장하지 않는다.
- **복수 출처 보존** — 같은 항목의 서로 다른 출처 값을 보존할 수 있는 구조를 유지한다(불일치 표현 가능).
- **검수 상태 표현** — 데이터의 신뢰 수준(예: 검증됨 / 검수중)을 표현 가능하게 둔다.
- **목 우선** — 실데이터·API 키가 없으면 인터페이스 + 목(mock)으로 골격을 세우고 연동 지점을 `TODO`로 남긴다.
- **가정 명시** — 모호한 지점은 임의로 채우지 말고 `ASSUMPTION`으로 표시한 뒤 진행한다(원칙은 예외).
- **비밀 분리** — 키·토큰은 하드코딩하지 않고 환경변수로 분리한다.
- **작은 단위** — 한 변경/PR은 한 가지 일을 한다.
- **공개 가능 상태 유지** — 라이선스 비호환 의존성은 도입 전 표면화한다.
- **떠나도 이어지게** — 결정 근거를 남겨 다른 사람이 인수할 수 있게 한다.
- **한국어로 작성** — 모든 산출물(문서·코드 주석·UI 문자열·커밋 메시지·반복 기록)은 한국어로 작성한다. 코드 식별자와 기술 용어는 원형을 유지한다.

## 5. 현재 기술 기반 (확장 가능)

> 이 절은 고정이 아니라 현재 선택이다. 더 나은 선택이 제안되면 0장 원칙 안에서 교체될 수 있다.

- **Frontend:** Next.js (React 기반), 정적 생성 우선. 비영리·무료 티어·서버비 최소화 전제.
- **AI:** LLM(개체 정합·불일치 분류) + RAG(인용 응답).
- **데이터:** 공공 API 수집 + 검수, 출처 메타데이터 추적, 정기 스냅샷(CSV/JSON) 릴리스.
- **소스(현재):** 열린국회정보 API, 공공데이터포털, 헌정회, 선거관리위원회, 뉴스 검색 API / RSS.
  새 소스 추가는 1장 게이트를 거친다.

## 6. 검토 관점 (Reviewer)

특정 체크리스트로 고정하지 않는다. 무엇을 만들든 Reviewer는 아래 **질문**으로 본다.

- 이 변경이 없던 사실을 만들어내지 않는가?
- 노출되는 모든 사실에 출처가 붙어 있는가?
- 근거가 없을 때 "모름"으로 처리되는가?
- 불일치가 병합되지 않고 드러나는가?
- 데이터 성격 분리가 지켜지는가?
- 타인 콘텐츠를 재호스팅하지 않는가?
- 공개 가능(라이선스·출처) 상태가 유지되는가?
- 확장 아이디어라면 1장 게이트를 통과했는가?

새로운 기능 유형이 생기면 그에 맞는 질문을 이 관점에 **추가**한다(축소하지 않는다).

---

## 7. 명령어

> 프로젝트 셋업 확정 후 이 절을 갱신한다. 아래는 골격이다.

```bash
npm install      # 의존성
npm run dev      # 개발 서버
npm run build    # 정적 빌드
npm test         # 테스트
npm run lint     # 린트
```

작업 종료 전 최소한 테스트와 린트를 실행하고 결과를 검토에 포함한다.
